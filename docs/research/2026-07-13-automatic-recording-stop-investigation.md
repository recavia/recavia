# ミーティング終了時の録音自動停止に関する調査

- 調査日: 2026-07-13
- 状態: 自動停止機能を削除
- 対象: macOS 上の Recavia によるマイク・システム音声同時録音

## 結論

今回確認した汎用 API だけでは、実環境のミーティング終了を誤検出なく判定できなかった。

`AVAudioEngineConfigurationChange` は Recavia のマイクエンジンやデバイス構成の変化を通知するもので、ミーティングアプリの退出を表すイベントではない。CoreAudio のプロセス単位の入力・出力状態も、検証したミーティング中に対象プロセスを公開しなかった。

誤停止による録音欠落を避けるため、ユーザー向けの自動停止設定と実行時の自動停止処理を削除した。手動停止、致命的なキャプチャ障害時の停止、録音データの保存処理は維持する。

## 検証環境

- Recavia は `./scripts/run-dev.sh` で起動した。
- このスクリプトは Debug バイナリを `Recavia.app` に格納し、`Recavia.entitlements` を付けてコード署名した後、バンドル内の実行ファイルを起動する。
- 親プロセスが Terminal であること自体は、CoreAudio が他プロセスの状態を公開するかどうかには影響しないと判断した。
- `swift run Recavia` や `.build/debug/Recavia` の直接実行は未署名なので、今回の権限切り分けには使用していない。

## 調査経緯

### 1. `AVAudioEngineConfigurationChange` の監視

当初の自動停止経路は、マイクの `AVAudioEngineConfigurationChange` を受け、エンジンが停止していた場合に録音停止を要求する設計だった。

実際のミーティング終了後も、ログにはマイクキャプチャ開始と最初のバッファ受信だけが記録され、次のログは発生しなかった。

```text
[MicrophoneCapture] Received microphone engine configuration change notification
```

このため、ミーティング終了時に自動停止経路へイベントが届いていないことが分かった。

### 2. 外部プロセスのマイク入力状態

CoreAudio の次のプロパティを使用し、Recavia 自身の PID を除外して、録音開始時に入力ストリームを使用している外部プロセスを追跡した。

- `kAudioHardwarePropertyProcessObjectList`
- `kAudioProcessPropertyPID`
- `kAudioProcessPropertyBundleID`
- `kAudioProcessPropertyIsRunningInput`

実機ログ:

```text
[ExternalMicrophoneActivity] External microphone activity monitor was not armed because no other input process was active
```

ミーティング中にもかかわらず、外部入力プロセスは 0 件だった。Recavia 自身のマイク利用を除外できる点は有効だったが、対象ミーティングアプリまたはブラウザが CoreAudio のプロセス入力状態として現れなかったため、この方式は利用できなかった。

### 3. 外部プロセスの音声出力状態

入力状態を使用せず、録音開始時から `kAudioProcessPropertyIsRunningOutput` のみを追跡する方式も検証した。これは音量の無音判定ではなく、プロセスが CoreAudio の出力ストリームを実行中かどうかを確認する方式である。

実機ログ:

```text
[ExternalAudioOutput] External audio output monitor was not armed because no other output process was active
```

入力と同様に、ミーティング中でも外部出力プロセスは 0 件だった。プロセス一覧そのものの取得に失敗した場合は別のエラーログを出す実装だったため、このログは API 呼び出しの失敗ではなく、条件に一致するプロセスが存在しなかったことを示す。

## 採用しなかった方式

### オーディオデバイス全体の使用状態

`kAudioDevicePropertyDeviceIsRunningSomewhere` は、Recavia 自身のマイクキャプチャにも反応する。Recavia が録音中である限りマイクデバイスは使用中になるため、ミーティングアプリの終了判定には使えない。

### システム音声の無音時間

ScreenCaptureKit が渡す音声バッファの振幅やバッファ停止時間を監視する案がある。ただし、次の状況をミーティング終了と区別できない。

- 会議中に参加者全員が発話していない
- 相手側がミュートしている
- 発表資料を黙って確認している
- ブラウザや OS が無音バッファを継続または停止する実装差

長い猶予時間を設けても誤停止の可能性は残り、短くすると録音欠落の危険が高くなる。この調査では実装を見送った。

### アプリの起動・終了

Zoom、Teams、Slack などはミーティング退出後もアプリ自体が起動し続ける。ブラウザ会議ではブラウザのプロセスも継続するため、プロセス終了はミーティング終了を表さない。

### ウィンドウタイトル

Google Meet や Zoom の会議ウィンドウをタイトルで検出できる場合があるが、次の制約がある。

- ブラウザ、言語、会議サービスの UI 更新でタイトルが変化する
- バックグラウンド、最小化、別 Space で取得結果が変わる可能性がある
- Screen Recording または Accessibility 権限の影響を受ける
- ネイティブアプリでは退出後も同じウィンドウが残る場合がある

単独の停止条件としては採用しなかった。

## 今後再検討する場合の候補

自動停止を再導入する場合は、単一の汎用シグナルではなく、複数条件を組み合わせてサービス別に検証する。

1. ScreenCaptureKit の音声バッファについて、コールバック間隔、ピーク値、RMS、完全なゼロ値を診断ログへ記録する。
2. 一度有効なシステム音声を観測した後だけ無音監視を有効化し、十分に長い猶予時間を設ける。
3. 会議ウィンドウの消失やタイトル変化を補助条件にし、音声無音だけでは停止しない。
4. Google Meet、Zoom、Microsoft Teams など、対象サービスごとにミュート、長い沈黙、退出、アプリ継続のテストマトリクスを作る。
5. 自動停止前に通知またはカウントダウンを表示し、ユーザーが停止を取り消せる設計を検討する。

最低限、次のシナリオで誤停止と停止漏れの両方を確認する必要がある。

| シナリオ | 期待結果 |
|---|---|
| 通常の発話中 | 継続 |
| 参加者全員が長時間ミュート | 継続 |
| システム音声が一時的に途切れる | 継続 |
| 会議から退出し、アプリは起動したまま | 停止候補 |
| 会議終了後に別アプリが音声を再生 | 誤停止・停止漏れを評価 |
| ブラウザの別タブで音声を再生 | 誤停止・停止漏れを評価 |

## 残した診断機能

- デバッグ設定から独立ウィンドウで開くアプリケーションログ
- ログの検索、テキスト選択、表示中ログのコピー
- マイクキャプチャ開始、フォーマット、最初のバッファ、構成変更のログ
- 独立したマイク・音声認識テストウィンドウ

ログには音声内容、文字起こし本文、資格情報などのユーザーデータを含めない。

## 削除した機能

- 一般設定の「録音を自動停止」トグル
- `automaticallyStopRecordingWhenMicrophoneStops` の設定参照
- マイク構成変更を録音停止要求として ViewModel へ伝播する経路
- 非致命的なマイク停止を設定に応じて録音停止へ変換する判定
- 上記判定専用のテスト

既存の UserDefaults に同名のキーが残っていても参照されず、録音動作には影響しない。
